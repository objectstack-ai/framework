# Deployment & Multi-Project

The same `apps/server` binary boots into one of two shapes. Which one you get
depends only on the environment variables you set ΓÇö the code and Docker image
are identical.

| Shape                        | Control plane           | Projects    | Hostname routing | Typical use                          |
| ---------------------------- | ----------------------- | ----------- | ---------------- | ------------------------------------ |
| **single**                   | ΓÇö (single kernel)       | 1           | Γ¥î                | OSS single-project deploy            |
| **multi-project**            | `OBJECTSTACK_DATABASE_URL` | 1 ΓÇô N    | Γ£à                | Framework Studio dev, self-hosted multi-project, SaaS |

Both shapes reuse:

- `KernelManager` + `DefaultProjectKernelFactory` for per-project kernels
- `DefaultEnvironmentDriverRegistry` for hostname ΓåÆ project resolution
- `createControlPlanePlugins()` (multi-project only) to install
  tenant / auth / security / audit / package plugins onto the control DB.

---

## Shape 1 ΓÇö single

```bash
pnpm --filter @objectstack/server dev
# ΓåÆ http://localhost:3000
```

One `ObjectKernel`, every plugin from `apps/server/objectstack.config.ts`, no
per-project isolation. This is what you want on a laptop or for an OSS
single-app install.

Leave `OBJECTSTACK_MULTI_PROJECT` unset (or set it to `false`) to use this
shape.

## Shape 2 ΓÇö multi-project

```bash
OBJECTSTACK_MULTI_PROJECT=true \
OBJECTSTACK_DATABASE_URL=file:./.objectstack/data/control.db \
AUTH_SECRET=$(openssl rand -hex 32) \
pnpm --filter @objectstack/server dev
```

- Control-plane tables (`sys_project`, `sys_project_credential`, ΓÇª) live in
  the database configured by `OBJECTSTACK_DATABASE_URL`. Use `file:...` for a
  local SQLite control plane, or `libsql://...` / `https://...` for a remote
  Turso/libSQL control plane.
- Projects are created through Studio / `POST /api/v1/cloud/projects`. Each
  project binds its own driver (sqlite / libsql / turso / postgres).
- Incoming requests are routed to per-project kernels by the `Host` header
  (matched against `sys_project.hostname`), or by `X-Project-Id`, or by the
  authenticated session's `activeProjectId`.

### Configuring a project driver

When calling `POST /api/v1/cloud/projects`, pass `driver` as one of the
registered drivers: `memory`, `sqlite`, `turso` / `libsql`, or `postgres`.
The factory switch lives in
`packages/runtime/src/project-kernel-factory.ts::createDriver`.

For a remote shared control plane, only the URL changes:

```bash
OBJECTSTACK_MULTI_PROJECT=true \
OBJECTSTACK_DATABASE_URL=libsql://control.turso.io \
OBJECTSTACK_DATABASE_AUTH_TOKEN=... \
AUTH_SECRET=$(openssl rand -hex 32) \
pnpm --filter @objectstack/server dev
```

Multiple Server replicas can share the same remote `sys_*` schema;
`DefaultEnvironmentDriverRegistry`'s per-replica cache is coherent within its
TTL (5 min default), and credential rotation fires
`envRegistry.invalidate(projectId)` so stale entries drop out within one TTL.

## Hostname routing precedence

`HttpDispatcher.resolveEnvironmentContext` tries these signals in order and
stops at the first match:

1. `/projects/:projectId/...` URL segment.
2. `Host` header ΓåÆ `sys_project.hostname` (via `envRegistry.resolveByHostname`).
3. `X-Project-Id` header.
4. Session `activeProjectId`.
5. Session `activeOrganizationId` ΓåÆ that org's default project.

Control-plane routes (`/auth`, `/cloud`, `/discovery`, `/health`) skip
resolution and always hit the control-plane kernel.

---

## Cross-subdomain session cookies

Host-based routing means Studio (say `studio.example.com`) and a project
subdomain (`acme.example.com`) are separate origins. Set
`OBJECTSTACK_COOKIE_DOMAIN=.example.com` to opt in to better-auth's
cross-subdomain cookie mode. In production the session cookie is also scoped
`secure` automatically.

## Deploying to Fly.io

ObjectStack ships a single Dockerfile (`apps/server/Dockerfile`) that works
for both Fly.io and bare Docker. The repo also ships a ready-to-use
`apps/server/fly.toml`.

```bash
# From the repo root
fly launch --copy-config --name objectstack-server \
           --dockerfile apps/server/Dockerfile \
           --no-deploy

# Persistent volume for the local-SQLite control DB
fly volumes create objectstack_data --size 3 --region iad

# Secrets (never commit!)
fly secrets set \
  AUTH_SECRET=$(openssl rand -hex 32) \
  OBJECTSTACK_COOKIE_DOMAIN=.objectstack.app \
  TURSO_ORG_NAME=... \
  TURSO_API_TOKEN=...

fly deploy

# Wildcard cert for per-project subdomains
fly certs add "*.objectstack.app"
# Follow the DNS instructions Fly prints (_acme-challenge TXT + A/AAAA)
```

### Fly tuning

`OBJECTSTACK_KERNEL_CACHE_SIZE` and `OBJECTSTACK_KERNEL_TTL_MS` control the
per-project kernel LRU. The defaults in `fly.toml` (50 kernels, 30 min TTL)
fit a `shared-cpu-2x / 1GB` machine comfortably. Bump both proportionally if
you upgrade the VM.

## Deploying with Docker (self-hosted)

```bash
docker build -f apps/server/Dockerfile -t objectstack/server:local .

docker run --rm -p 3000:3000 \
  -v $(pwd)/data:/data \
  -e OBJECTSTACK_MULTI_PROJECT=true \
  -e OBJECTSTACK_DATABASE_URL=file:/data/control.db \
  -e AUTH_SECRET=$(openssl rand -hex 32) \
  objectstack/server:local

curl http://localhost:3000/api/v1/health
```

Put a reverse proxy (Caddy / Traefik / Nginx) in front for wildcard TLS and
point a wildcard DNS (`*.example.com`) at it. The container does the rest.

## Environment variable reference

| Variable                          | Shape(s)                  | Purpose                                                                          |
| --------------------------------- | ------------------------- | -------------------------------------------------------------------------------- |
| `OBJECTSTACK_MULTI_PROJECT`       | all                       | `true` enables the multi-project control plane. Unset/false runs single-project mode. |
| `OBJECTSTACK_DATABASE_URL`        | all                       | In `single`, the project business DB URL. In `multi-project`, the control-plane DB URL (`file:`, `libsql://`, or `https://`). |
| `OBJECTSTACK_DATABASE_AUTH_TOKEN` | all                       | Auth token for `OBJECTSTACK_DATABASE_URL` when using libSQL/Turso.               |
| `TURSO_ORG_NAME` + `TURSO_API_TOKEN` | multi-project (any)    | Used by `TursoProjectDatabaseAdapter` to provision per-project databases.        |
| `OBJECTSTACK_KERNEL_CACHE_SIZE`   | multi-project (any)       | LRU size for per-project kernels (default 32, fly.toml defaults to 50).          |
| `OBJECTSTACK_KERNEL_TTL_MS`       | multi-project (any)       | Idle eviction TTL in ms (default 900000, fly.toml defaults to 1800000).          |
| `OBJECTSTACK_ENV_CACHE_TTL_MS`    | multi-project (any)       | TTL for the envRegistry's hostname/id cache (default 300000).                    |
| `OBJECTSTACK_COOKIE_DOMAIN`       | multi-project (any)       | Shared cookie domain for cross-subdomain sessions (e.g. `.objectstack.app`).     |
| `AUTH_SECRET`                     | all                       | Better-Auth session secret (ΓëÑ 32 chars).                                         |
| `PORT` / `HOST`                   | all (node-entry)          | HTTP bind. Defaults to `3000` / `0.0.0.0`.                                       |

## Smoke test (multi-project)

```bash
# 1. Create a project
curl -X POST http://localhost:3000/api/v1/cloud/projects \
  -H 'Content-Type: application/json' \
  -d '{ "organization_id": "<org-id>", "display_name": "Demo", "driver": "sqlite" }'
# ΓåÆ { id: "<pid>", status: "provisioning", hostname: "<org-slug>-<short>.objectstack.app", ... }

# 2. Wait for status=active (Studio also polls this)
curl http://localhost:3000/api/v1/cloud/projects/<pid>

# 3. Call the project-scoped API via Host header
curl -H "Host: <hostname>" http://localhost:3000/api/v1/meta/objects
# ΓåÆ returns objects registered in the project's own sys_metadata, not the control plane
```

If step 3 returns control-plane objects instead of the project's own, the
host-based routing did not kick in ΓÇö check that `envRegistry` is wired (see
`apps/server/server/bootstrap.ts`) and that `sys_project.hostname` was
populated.
